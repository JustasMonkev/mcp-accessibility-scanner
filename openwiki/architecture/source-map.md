# Source map

Use this page to locate ownership before editing. Source of truth is TypeScript under `src/`; `lib/` is generated by `npm run build`.

## Root surfaces

| Path | Responsibility | Change implications |
|---|---|---|
| `package.json` | package identity, bins, Node engine, scripts, dependencies, publish files | Preserve package and binary compatibility; build/test scripts are operational contracts. |
| `cli.js` | published executable shim into `lib/program.js` | Normally unchanged; source behavior belongs in `src/program.ts`. |
| `index.js`, `index.d.ts`, `config.d.ts` | public package API shims/types | Keep aligned when changing library/config surface. |
| `README.md` | primary user install, CLI, configuration, and tool documentation | Update for any user-visible behavior. |
| `SKILL.md` | agent-oriented interactive CLI recipes, including Electron | Update when tool usage or fallback workflows change. |
| `Dockerfile`, `docker-compose.yaml` | production image and local HTTP service | Re-run Docker smoke for changes. |
| `vitest.config.ts` | test runtime and coverage policy | Large exclusions mean thresholds do not represent all runtime layers. |

## Runtime core

| Path | Owns |
|---|---|
| `src/program.ts` | Commander CLI, mode dispatch, interactive REPL, process cleanup. |
| `src/index.ts` | programmatic `createConnection()` API. |
| `src/config.ts` | defaults, config/env/CLI merge, validation, output path resolution. |
| `src/browserServerBackend.ts` | MCP backend adapter, tool validation/execution, context creation. |
| `src/browserContextFactory.ts` | persistent, isolated, remote, CDP attach, and CDP launch strategies. |
| `src/context.ts` | lazy browser context, tabs, network routing, tracing, global disposal. |
| `src/tab.ts` | Playwright page wrapper, status/artifact collection, modal state, snapshots, refs. |
| `src/response.ts` | markdown/images/resource links/structured-content serialization and progress. |
| `src/sessionLog.ts` | optional session/input/response recording. |
| `src/tools.ts` | complete tool registry, capability filtering, server instructions. |

Typical startup trace: `cli.js` → generated `lib/program.js` from `src/program.ts` → `src/mcp/server.ts` → `BrowserServerBackend` → lazy `Context` → selected `BrowserContextFactory` → `Tab`.

## Generated and incidental directories

- `lib/`: compiled source; regenerate, do not hand-edit.
- `coverage/`: coverage output.
- `output/`: reports/screenshots and Docker-mounted artifacts.
- `graphify-out/`: generated analysis output, not part of runtime ownership found during this pass.
- `node_modules/`: dependencies.
- `.claude/`: direct MCP harness and local automation support.

## Change-impact shortcuts

- Adding a tool normally touches its `src/tools/*.ts` module, `src/tools.ts`, focused tests, and user docs. Preserve tool names unless intentionally breaking clients.
- Adding config requires source/public types, CLI/env parsing as applicable, merge semantics, validation, README, and config/program tests.
- Changing response/report shape requires `src/response.ts` or an audit tool, serialization tests, and consideration for both markdown and structured consumers.
- Changing browser connection behavior requires factory tests plus at least one integration/harness path; attached CDP targets have different ownership from launched contexts.
- Changing HTTP requires session, routing, Host/Origin, readiness, and heartbeat tests—not only happy-path MCP calls.
